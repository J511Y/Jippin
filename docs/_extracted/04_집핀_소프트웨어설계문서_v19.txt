목차
1. 문서 개요
1.1 목적과 범위
본 소프트웨어 설계문서(SDD)는 집핀 서비스를 어떤 논리 모듈로 분해하며, 각 모듈이 어떤 책임을 지고, 모듈끼리 어떤 인터페이스로 결합하는지를 정의한다. 본 문서는 “무엇을 만들 것인가(What)”가 아니라 “어떻게 분해할 것인가(How to decompose)”에 답한다.
핵심 원칙 — 모듈 분해는 코드가 물리적으로 어디서 실행되는가(프론트엔드/백엔드)가 아니라, 어떤 책임을 지는가(논리 도메인)를 기준으로 한다. 프론트엔드·백엔드 구분은 논리 모듈을 배포 단위에 매핑하는 §2.4와 각 모듈의 ‘배포 매핑’ 필드에서만 다룬다.
범위 외 항목 — 본 문서는 다음을 다루지 않는다. 해당 항목은 부록 A의 정본 분담표를 참조한다.
기술 스택 선정 근거, 라이브러리 버전 (→ 기술명세서)
API 엔드포인트 상세 명세, 요청·응답 페이로드 (→ 기술명세서)
물리 DB 스키마(컬럼 타입·제약·인덱스) (→ 기술명세서)
기능별 수용 기준, 사용자 플로우 (→ 기능명세서)
코드 레벨 구현물 — TypeScript 인터페이스, 컴포넌트 Props 시그니처, Zustand 스토어, Python 클래스, 디렉토리 트리 (→ 각 리포지토리 README·소스 코드)
배포 산출물, 운영 정책 수치(로그 보존, Rate Limit, 캐싱 TTL) (→ 기술명세서·운영 가이드)

2. 시스템 아키텍처
2.1 아키텍처 원칙
책임 기준 모듈 분해 — 모듈은 물리 배포 위치가 아니라 도메인 책임을 기준으로 분해한다. 하나의 논리 모듈은 여러 배포 단위에 걸칠 수 있다.
단방향 계층 의존 — Presentation, Application, Data 3개 논리 계층 간 의존은 단방향이며 역방향 호출을 허용하지 않는다.
AI 처리는 동기 — 사용자 체감 응답시간(< 30s) 목표 하에 비동기 큐 없이 워커 멀티프로세싱으로 처리한다.
AI 모델 추상화 — 프로바이더 교체(OpenAI ↔ 자체 호스팅)가 자유롭도록 LangChain 추상 인터페이스로 격리한다.
법적 판단 분리 — RULE 모듈을 AI 파이프라인 바깥에 두고, AI 출력은 ‘공통 판단 스키마’ 컨트랙트로만 RULE 모듈에 진입한다.
2.2 3-Layer 논리 구조
집핀은 Presentation · Application · Data 3개 논리 계층으로 구성된다. ‘계층’은 배포 단위가 아니라 책임의 추상화 수준을 가리킨다. 각 계층은 아래 계층에만 의존한다.

--- TABLE ---
논리 계층 | 주요 책임 | 대표 배포 단위
Presentation | 사용자 인터랙션, A2UI 렌더링, 도면 오버레이 표시, OAuth 콜백 수신 | Next.js (App Router) + Nginx
Application | 요청 라우팅, 모듈 오케스트레이션, AI 파이프라인, 법규 판단, 외부 API 어댑팅 | FastAPI (Gunicorn/Uvicorn)
Data | 사용자·세션·도면·리포트 영속화, 캐시, 객체 저장 | PostgreSQL + Redis + S3
--- /TABLE ---

2.3 네트워크 토폴로지
Nginx가 모든 인바운드 트래픽을 종단하며 경로 기반 리버스 프록시로 분배한다.
/api/*  →  FastAPI   (Application 계층)
/*      →  Next.js   (Presentation 계층)
동일 EC2 인스턴스 내 컨테이너 네트워크 격리. PostgreSQL은 별도 EC2-DB로 분리한다. 인프라 구체 사양은 기술명세서 §7.2가 정본이다.
2.4 논리 모듈 ↔ 배포 단위 매핑
§3·§4가 정의하는 8개 논리 모듈은 아래와 같이 물리 배포 단위에 매핑된다. 하나의 논리 모듈이 Presentation·Application·Data에 걸쳐 분산될 수 있다 — 이는 모듈의 책임이 깨진 것이 아니라, 동일 책임이 사용자 인터랙션 측과 서버 처리 측에 나뉘어 구현되었음을 뜻한다.

주의 — 본 표는 모듈을 배포에 ‘투영’한 결과일 뿐이다. 설계의 1차 단위는 어디까지나 논리 모듈이며, 프론트엔드·백엔드라는 용어는 본 문서에서 이 표와 모듈별 ‘배포 매핑’ 필드 외에는 사용하지 않는다.
3. 모듈 분해 (Module Decomposition)
3.1 모듈 분해 개요와 구조도
집핀은 기능명세서 §2가 정의한 8개 도메인을 그대로 논리 모듈 경계로 채택한다. 각 모듈은 단일 책임을 가지며, 사용자 플로우 9단계는 이 모듈들이 정해진 순서로 협력한 결과다.
구조도 (Structure Chart) — CHAT 모듈이 사용자 응답과 세션 스키마 초안을 구성하고, 전체 절차 감시 에이전트(FLOW_GUARD)가 전체 절차의 충분성 판단과 보완 루프 제어를 독립 책임으로 수행한다. 본 도면의 화살표는 API 호출 계약이 아니라 대표 산출물 흐름과 오케스트레이션 방향을 요약한다. 실제 제공/소비 인터페이스 계약은 §5.1 및 그림 3을 정본으로 한다.
도면 처리 본류는 INPUT → MASK → AI → OVERLAY 순으로 흐른다. 이후 OVERLAY의 벽체 선택값과 CHAT의 부족 판단값 수집 결과가 CHAT/session 계층에서 CommonJudgmentSchemaDraft로 병합되고, FLOW_GUARD가 충분성·충돌·고위험 여부를 평가한다. FLOW_GUARD가 PROCEED_RULE을 반환한 경우에만 RULE이 완성된 단일 스키마를 평가하고, REPORT는 RuleEvalResult를 바탕으로 리포트와 예상 견적을 산출한다.
[그림 3] 모듈 연계도 — 책임 배치와 대표 산출물 흐름(직접 API 계약도 아님)
3.2 모듈 분해 근거 (Design Rationale)

--- TABLE ---
분해 결정 | 근거
분해 축으로 8개 도메인을 채택 | 기능명세서가 이미 AUTH/INPUT/MASK/AI/OVERLAY/CHAT/RULE/REPORT 도메인으로 요구사항을 구획했다. 동일 경계를 설계에서도 유지하면 요구사항→설계 추적성이 1:1로 확보된다.
MASK를 AI에서 분리 | 수치 마스킹은 ‘도면 자산 유출 방지’라는 독립 책임이며 AI 분석 정확도와 무관하다. 분리하면 마스킹 정책을 AI 모델 교체와 독립적으로 진화시킬 수 있다.
RULE을 AI에서 분리 | 법적 판단은 결정성·감사 가능성이 필요하다. AI 출력, 오버레이 선택값, 대화형 보완값은 CHAT/session 계층에서 공통 판단 스키마로 병합되고, RULE은 완성된 스키마만 입력으로 수신한다. 따라서 AI 모델·오버레이 UI·대화 UX 변경이 법규 판단 로직에 직접 파급되지 않는다.
OVERLAY를 CHAT에서 분리 | 도면 위 좌표 기반 인터랙션은 채팅 메시지 인터랙션과 입력 양식·상태 모델이 다르다. 분리하면 캔버스 렌더링 기술 선택이 채팅 모듈에 영향을 주지 않는다.
CHAT을 오케스트레이터로 지정 | 사용자 플로우 9단계가 모두 채팅 세션 안에서 진행된다. 흐름 제어 책임을 CHAT에 집중시켜 다른 모듈은 순수 도메인 처리에만 집중하게 한다.

4. 모듈별 상세 설계
8개 논리 모듈을 동일한 7개 필드 템플릿으로 기술한다. 본 장은 모듈의 책임과 인터페이스 계약까지만 다루며, 클래스명·함수 시그니처·디렉토리 구조 등 코드 레벨 표현은 각 모듈의 ‘코드 정본’이 담당한다.
제공/요구 인터페이스의 오퍼레이션은 ‘오퍼레이션명(입력) → 출력 — 의미’ 형식의 계약 수준으로만 기술한다.
4.1 AUTH — 인증·계정 모듈

--- TABLE ---
책임 (Responsibility) | 소셜 OAuth 기반 사용자 식별과 JWT 세션 수명주기(발급·갱신·무효화)를 단독으로 책임진다.
제공 인터페이스 (Provided) | authenticateOAuth(provider, authCode) → {accessToken, refreshToken, userContext} — OAuth 콜백 코드를 검증하여 사용자를 식별·자동가입하고 토큰을 발급한다. / refreshSession(refreshToken) → {accessToken} — 만료된 Access Token을 사용자 인지 없이 재발급한다. / resolveCurrentUser(accessToken) → UserContext — 모든 보호 라우터에 사용자 컨텍스트를 주입한다(인가 의존성). / revokeSession(accessToken) → void — 로그아웃·관리자 강제 만료 시 토큰을 무효화한다.
요구 인터페이스 (Required) | 외부 OAuth 프로바이더 — Authorize / Token 교환 / UserInfo 조회. / users 엔티티 영속화(Data 계층 upsert). / JWT 블랙리스트 저장소(Redis).
데이터 계약 (Data Contract) | UserContext { user_id, email, role(user|admin) }. 비밀번호는 보관하지 않는다.
오류·예외 | OAUTH_FAILED — 외부 인증 실패. / TOKEN_INVALID / TOKEN_EXPIRED — Access Token 검증 실패. / REFRESH_EXPIRED — Refresh Token 만료. 재로그인 유도.
배포 매핑 (논리 계층) | Presentation(OAuth 콜백·토큰 인터셉터) + Application(인증 미들웨어·auth 라우터) + Data(users·블랙리스트).
코드 정본 (Source of Truth) | 기능명세서 §2.1(AUTH-001~002). 토큰 유효기간 등 수치는 기술명세서 §5.1. 코드: frontend lib/auth.ts, backend middleware/auth.py·routers/auth.py.
--- /TABLE ---

4.2 INPUT — 입력·도면조회 모듈

--- TABLE ---

데이터 계약 (Data Contract) | 예상 견적 결과(EstimateResult) { permit_agency_fee_estimate(행위허가 대행 비용), fire_panel_estimate(방화판 설치 비용), fire_glass_estimate(방화유리 설치 비용), total_range(합산 범위), assumptions(산정 전제), policy_version(정책 버전), variance_notes(변동 사유), consultation_required(상담 권장 여부) }. 단가 정책 예시(v1): 행위허가 대행 33만원 / 방화판 설치 가로 길이 × 2만원/m / 방화유리 설치 유리 자재비 + 시공비 5만원. 특허 출원 설명에서는 특정 단가를 고정하지 않고 정책에 따라 변경 가능한 산정식·단가표로 표현하며, 모든 단가는 운영 정책에 따라 변경 가능하다.
오류·예외 | 정책 단가표 누락(PRICING_POLICY_MISSING) — 단가표 미연결 시 견적 항목을 생략하고 상담을 권고. 견적 범위 초과(ESTIMATE_OUT_OF_RANGE) — 산정 결과가 정책 허용 범위를 초과하면 보수적 범위로 클립하고 사유를 기록.
배포 매핑 (논리 계층) | Application 전용. REPORT 모듈 하위 산출부로 결합되며 별도 배포 단위로 분리되지 않는다.
코드 정본 (Source of Truth) | 기능명세서 §2.9(REPORT-003). 코드: backend services/report_generator/estimate.

8. 횡단 관심사 (Cross-Cutting Concerns) 설계
본 장은 ‘어디서 어떻게 처리하는가’의 설계 의도만 다룬다. 수치(토큰 유효기간·로그 보존·Rate Limit·TTL)는 기술명세서·운영 정책이 정본이다.
8.1 인증·인가 설계
인증 책임 위치 — AUTH 모듈의 인증 미들웨어에서 JWT를 검증한다. 다른 모듈의 라우터는 인가 의존성 주입으로 사용자 컨텍스트만 받는다.
권한 모델 — RBAC 2단계(user / admin). /admin/* 라우터는 별도 관리자 게이팅을 적용한다.
세션 격리 — 모든 자원 접근 라우터에서 session.user_id == current_user.id 사전 검증을 일관 적용한다.
토큰 갱신 — Presentation의 HTTP 인터셉터가 401 응답을 감지하여 refreshSession 후 원본 요청을 자동 재시도한다.
토큰 무효화 — Redis JWT 블랙리스트로 로그아웃·강제 만료를 처리한다.
토큰 유효기간·서명 알고리즘 등 구체 수치는 기술명세서 §5.1 참조(부록 A에 정합성 이슈 기재).
8.2 에러 처리 전략
에러 도메인 분류 — 클라이언트 오류(4xx), 외부 시스템 오류(502/504), AI 신뢰도 부족(422), 내부 오류(500) 4개 그룹으로 분리한다.
처리 위치 — 비즈니스 예외는 모듈 내부에서 도메인 예외(ZippinException 계열)로 raise하고, 공통 예외 핸들러가 표준 응답 포맷으로 변환한다.
AI 분석 실패 — AI 모듈의 각 단계는 timeout·예외를 도메인 예외로 변환한다. 신뢰도 0.6 미만 시 ANALYSIS_LOW_CONFIDENCE로 재업로드를 권장한다.
응답 포맷 — error.{code, message, request_id, timestamp}를 표준으로 한다. detail은 개발 환경에서만 노출한다.
8.3 로깅 전략
구조화 로깅 — structlog로 JSON 로깅. request_id를 컨텍스트에 주입하여 모듈·외부 호출 전반에 전파한다.
로깅 지점 — 로깅 미들웨어에서 요청·응답 1줄, 모듈 내부에서 외부 호출·도메인 이벤트, 예외 핸들러에서 에러+스택트레이스.
PII 마스킹 — 채팅 메시지·주소·연락처는 로그 전 마스킹 단계를 거친다.
외부 송신 — ERROR 레벨은 Sentry, 운영 메트릭은 CloudWatch로 송출한다.
8.4 캐싱 전략
계층 위치 — 모든 캐시는 Redis 단일 인스턴스에 집중하되 키 네임스페이스(token:bl, addr:, seumteo:)로 도메인을 분리한다.
캐시 위치 — 외부 API 응답(주소·세움터)은 INPUT 모듈에서 데코레이터로 감싼다. 비즈니스 로직은 캐시 존재를 가정하지 않는다.
AI 모델 상주 — Mask2Former는 워커 부팅 시 1회 로드 후 메모리 상주한다(워커별 싱글톤).
정적 에셋 — Nginx + CDN. 애플리케이션 코드의 관심사가 아니다.
