1. 문서 개요
1.1 목적
본 문서는 집핀 서비스의 기능 요구사항을 개발·디자인·기획팀이 공통으로 참조할 수 있도록 상세히 기술한다. 각 기능의 입력·출력·처리 로직·예외 처리·수용 기준을 명확히 정의하여 후속 기술명세서 및 소프트웨어 설계문서 작성의 기반을 제공한다.
1.2 서비스 개요
집핀은 공동주택 비내력벽 철거(발코니 확장·공간 통합 등) 행위허가 사전검토 AI 서비스다. 비전문 사용자가 주소와 도면 정보만으로 철거 가능 여부, 필요 방화시설, 법령 근거를 확인할 수 있으며, 판단이 어려운 케이스는 상담·행위허가 의뢰 채널로 자연스럽게 연결된다.
1.3 핵심 사용자 플로우
전체 플로우는 9단계로 구성되며, 주소 입력 이후 모든 상호작용은 채팅(A2UI) 인터페이스 안에서 진행된다.

--- TABLE ---
단계 | 구분 | 내용
1 | 로그인 | 소셜 OAuth (카카오/구글/네이버) 로그인 필수
2 | 주소 입력 | 도로명/지번 주소 + 아파트명·동·호수·층수 입력 후 채팅 세션 생성
3 | 도면 확인 | 내부 DB 후보 조회(3만 건) → 사용자 선택 또는 직접 업로드
4 | 수치 마스킹 | 업로드 도면의 치수·면적 수치 자동 마스킹(OCR 기반)
5 | AI 분석 | Mask2Former(세그멘테이션+벽체 분류) + VLM(문맥 해석) 동기 처리
6 | 오버레이 확인 | 도면 위 객체 오버레이 표시 → 사용자 클릭 선택 및 간소 보정
7 | 대화 정보 수집 | LLM 에이전트가 부족 판단값을 생활 언어 Q&A·선택지 카드로 수집
8 | 룰 엔진 판단 | 법령 룰 엔진이 철거 가능 여부·필요 방화시설·행위허가 필요 여부 산출
9 | 결과 리포트 | 사전검토 리포트 + 상담 신청/행위허가 의뢰/재검토/링크 공유 CTA
--- /TABLE ---

1.4 전제 및 제약
비즈니스 모델: 완전 무료 (리드 수집 목적, 상담/행위허가 의뢰에서 수익 발생)
서비스 형태: 웹앱 반응형, 모바일 우선 (iOS Safari · Android Chrome 최신 2개 버전)
사용자 인증: 소셜 OAuth 필수. 비회원 사전검토 불가
AI 판단의 최종 법적 효력 없음 → 모든 결과 화면에 법적 고지 문구 노출 의무
최종 행위허가 권한은 관할 행정기관에 있으며 본 서비스는 사전 검토에 한정됨
2. 기능 상세 명세
2.1 [AUTH] 인증·계정 관리
AUTH-001  소셜 OAuth 로그인
사용자는 카카오·구글·네이버 계정으로 로그인해야 서비스를 이용할 수 있다.

--- TABLE ---
항목 | 내용
지원 프로바이더 | 카카오, 구글, 네이버
발급 토큰 | JWT Access Token (유효기간 1시간) + Refresh Token (유효기간 30일)
신규 가입 | 최초 OAuth 로그인 시 계정 자동 생성. 이름·이메일·프로필 이미지 수집
실패 처리 | OAuth 인증 실패 시 오류 메시지 표시, 재시도 유도
수용 기준 | OAuth 콜백 처리 후 3초 이내 채팅 진입 / JWT 검증 실패 시 로그인 리다이렉트
--- /TABLE ---

AUTH-002  JWT 세션 갱신
Access Token 만료 시 Refresh Token으로 자동 갱신하여 사용자 인지 없이 세션을 유지한다.

--- TABLE ---
항목 | 내용
갱신 조건 | API 호출 시 401 Unauthorized 응답 수신
갱신 흐름 | /auth/refresh 호출 → 새 Access Token 수신 → 원본 요청 재시도
Refresh 만료 | 로그아웃 처리 + 로그인 화면 이동 + 세션 만료 토스트 표시
--- /TABLE ---

2.2 [INPUT] 사용자 입력·도면 조회
INPUT-001  공동주택 식별 정보 입력
사용자는 주소 입력 화면에서 다음 정보를 입력한다. 완료 후 "사전검토 시작" 버튼을 누르면 채팅 세션이 생성된다.

--- TABLE ---
필드 | 필수 | 설명
도로명/지번 주소 | 필수 | 카카오/네이버 주소 API 연동, 자동완성 제공
아파트명 | 필수 | 주소 선택 시 자동 채워짐 (수정 가능)
동 | 필수 | 예: 101동
호수 | 필수 | 예: 1201호
층수 | 필수 | 숫자 입력 또는 호수에서 자동 추출
평형 | 선택 | 도면 후보 필터링. 미입력 시 전체 후보 제시
--- /TABLE ---

수용 기준: 필수 항목 미입력 시 다음 단계 진행 불가. 입력값은 정규화되어 도면 조회·룰 엔진에 전달.
INPUT-002  주소 기반 평면도 후보 조회
입력 정보를 기반으로 내부 평면도 DB(약 3만 건)에서 해당 세대의 평면도 후보를 조회한다.

--- TABLE ---
케이스 | 처리
후보 존재 | 썸네일 + 메타(평형, 타입) 채팅 카드로 제시 → 사용자 선택
후보 없음 | 도면 직접 업로드 안내 또는 상담 신청 분기
수용 기준 | 후보 신뢰도 함께 제공. 선택된 도면이 이후 분석의 입력이 됨
--- /TABLE ---

INPUT-003  도면 직접 업로드
사용자가 보유한 준공도면·평면도를 채팅 인터페이스에서 업로드한다.

--- TABLE ---
항목 | 내용
지원 형식 | JPG, PNG, PDF
최대 용량 | 파일당 20MB (정책 조정 가능)
업로드 직후 | 미리보기 표시. 업로드 도면이 있으면 DB 후보보다 우선 사용
보안 검사 | 파일 형식 검증 + 바이러스 스캔 통과 후 처리 진행
예외 처리 | 허용 형식 외 거부 및 오류 메시지 표시
--- /TABLE ---

INPUT-004  도면 이미지 자동 보정
업로드 이미지의 기울기·명도/대비·노이즈·회전을 자동 보정하여 AI 분석 정확도를 높인다.
보정 전/후 비교 미리보기 제공
보정 실패 시 원본 이미지 사용 + 사유 로깅
INPUT-005  도면 내 텍스트 OCR 추출
도면에서 호수·평형·실명 등 텍스트를 OCR로 추출하여 공통 판단 스키마에 연결한다. OCR 추출은 수치 마스킹 이전에 수행한다.
핵심 라벨(실명, 호수) 인식률 80% 이상 목표
2.3 [MASK] 도면 수치 마스킹
MASK-001  업로드 도면 수치 자동 마스킹
사용자가 업로드한 도면에서 치수·단위·면적·좌표 수치 영역을 자동 검출하고, 가림 처리하여 사용자에게 회신한다. 이 이미지가 이후 AI 분석의 입력으로 사용된다.

--- TABLE ---
항목 | 내용
검출 대상 | 치수 숫자(mm/cm/m), 면적 표기(㎡), 좌표 수치, 배율 표시
마스킹 방식 | 해당 영역에 단색(흰색/회색) 사각형 오버레이 또는 모자이크 처리
결과 반환 | 마스킹된 도면 이미지를 채팅 버블로 표시 + S3 저장
목적 | (1) 도면 자산 유출 방지 (2) 사용자 신뢰 신호
판단 영향 | 수치보다 공간 인접 관계·벽체 유형이 판단 기준이므로 마스킹이 정확도에 영향 없음
수용 기준 | 업로드 후 10초 이내 마스킹 완료 + 미리보기 제공 / 실패 시 원본 사용 + 로그
--- /TABLE ---

2.4 [AI] 도면 해석 AI
AI-001  Mask2Former 세그멘테이션
마스킹된 평면도 이미지를 Mask2Former 모델에 입력하여 공간 영역 세그멘테이션 및 벽체 유형 분류 결과를 얻는다.

--- TABLE ---
항목 | 내용
역할 1: 영역 세그멘테이션 | 발코니·거실·주방·침실·복도 등 공간별 마스크 영역 생성
역할 2: 벽체 유형 분류 | 비내력벽 후보 / 내력벽 후보 분류
출력 형식 | 객체별 마스크 좌표 + 클래스 레이블 + 신뢰도 점수 (JSON)
정확도 목표 | 주요 클래스 mAP 80% 이상
신뢰도 임계값 미만 | "추가 확인 필요" 표시 → 사용자 재확인 또는 판단 보류
--- /TABLE ---

AI-002  VLM 도면 문맥 해석
OpenAI Vision API를 활용하여 도면 텍스트 라벨·공간명·애매한 경계를 해석한다. LangChain으로 추상화하여 프로바이더 교체가 가능하다.

--- TABLE ---
항목 | 내용
역할 | Mask2Former 결과 레이블 보완, 공간 명칭 확인, 경계 모호 영역 자연어 해석
입력 | 마스킹된 도면 이미지 + Mask2Former JSON + 사용자 발화 컨텍스트
출력 | 공간 객체 후보 리스트 (이름, 좌표, 유형, 신뢰도)
프로바이더 | 초기: OpenAI GPT-5.4-mini. LangChain 기반 추상화로 교체 가능
--- /TABLE ---

AI-003  복수 모델 결과 정합성 검증 및 공통 판단 스키마 정규화
두 모델의 결과를 비교하여 정합성을 검증하고, 공통 판단 스키마(JSON)로 정규화한다.

--- TABLE ---
케이스 | 처리
결과 일치 | 신뢰도 상향 → 확정 처리
결과 불일치 | 룰 기반 검증 → 해결 불가 시 사용자 재확인 또는 판단 보류
--- /TABLE ---

공통 판단 스키마 주요 필드:

--- TABLE ---
필드 | 내용
space_objects | 공간 객체 목록 (id, label, type, mask_coords, confidence, source_engine)
wall_objects | 벽체 목록 (id, wall_type: non_load_bearing|load_bearing, confidence, coords)
target_wall | 사용자가 선택한 철거 희망 비내력벽 ID
target_space | 사용자가 지정한 변경 희망 공간 ID
judgment_values | 룰 엔진 입력 변수 (층수, 스프링클러, 대피공간, 계단실 수, 창호 형태 등)
schema_version | 스키마 버전 문자열 (예: "1.0.0")
--- /TABLE ---

2.5 [OVERLAY] 오버레이 UI
OVERLAY-001  객체 후보 오버레이 표시
AI 분석 결과를 평면도 이미지 위에 반투명 색상 마스크와 레이블로 오버레이하여 채팅 인터페이스 내 인터랙티브 이미지로 표시한다.

--- TABLE ---
항목 | 내용

구현 기술 | Canvas 기반 라이브러리 (Fabric.js / Konva.js / SVG 중 테스트 후 결정)
모바일 대응 | 핀치 줌, 스와이프 스크롤 지원
수용 기준 | AI 분석 완료 후 3초 이내 오버레이 렌더링
--- /TABLE ---

OVERLAY-002  비내력벽 후보 클릭 선택
사용자는 오버레이된 비내력벽 후보 중에서 철거 희망 벽체를 클릭으로 선택한다.
단일 또는 복수 선택 지원. 선택 시 강조 표시. 선택 해제 가능
선택 결과는 공통 판단 스키마의 target_wall에 기록
2.6 [CHAT] LLM 대화형 에이전트
CHAT-001  채팅 세션 생성
주소 입력 완료 시 고유 채팅 세션(session_id: UUID v4)이 생성된다. 이후 모든 상호작용이 이 세션 안에서 진행된다.
세션 만료: 마지막 활동 후 24시간. 만료 1시간 전 경고
사용자별 과거 세션 목록 조회 가능 (마이페이지)
CHAT-002  A2UI 채팅 인터페이스 컴포넌트
시스템은 텍스트 메시지와 구조화된 선택지 카드를 혼합하여 제공한다.

--- TABLE ---
UI 컴포넌트 | 사용 시점
텍스트 버블 | 시스템 안내 메시지, 사용자 자유 발화 답변
선택지 버튼 | YES/NO 또는 2~5개 보기 제시 (예: 스프링클러 여부)
이미지 카드 | 도면 후보 썸네일, 마스킹 결과 미리보기
인터랙티브 오버레이 | 객체 선택·보정 필요 시 도면 오버레이 뷰 삽입
파일 업로드 버튼 | 도면 업로드 안내 시점
리포트 카드 | 최종 사전검토 결과 표시 + CTA 버튼 포함
--- /TABLE ---

CHAT-003  부족 판단값 식별 및 생활 언어 질의 생성
법령 판단에 필요한 변수 중 미확정 항목을 식별하고 생활 언어 질문 또는 시각적 선택지로 변환한다.

--- TABLE ---
판단 변수 | 전문 용어 | 생활 언어 질문 예시
sprinkler_coverage | 스프링클러 살수 범위 | "발코니 천장에 스프링클러가 있나요?"
exit_space_exists | 대피공간 유무 | "비상시 옆집으로 탈출하는 칸막이가 있나요?"
staircase_count | 계단실 수 | "건물 엘리베이터(계단실)가 몇 개인가요?"

existing_expansion | 기확장 여부 | "이 공간이 이미 확장되어 있나요?"

2.7 [FLOW_GUARD] 전체 절차 감시 에이전트
FLOW_GUARD-001  판단값 충분성 평가
CHAT/session이 구성한 CommonJudgmentSchemaDraft, 사용자 응답 이력, AI 분석 결과, 오버레이 선택값, 누락 필드 목록을 입력받아 사전검토 리포트와 법령 평가를 진행하기에 충분한지 판단한다.

--- TABLE ---
항목 | 내용
입력 | CommonJudgmentSchemaDraft, 사용자 응답 이력, AI 분석 결과, 오버레이 선택값, 누락 필드 목록
평가 항목 | 필수 판단값 충족 여부, AI 신뢰도, 모델 간 결과 충돌 여부, 고위험 케이스 여부
출력 | 충분성 판단 결정(CompletionDecision). 4가지 값: 추가 질의 요청(ASK_MORE) / 도면 재확인 요청(REQUEST_OVERLAY_REVIEW) / 룰 평가 진행(PROCEED_RULE) / 보류·상담 전환(HOLD_OR_HANDOFF)
수용 기준 | 부족·충돌·고위험 케이스의 ≥95%가 보완 루프 또는 보류로 분류
--- /TABLE ---

FLOW_GUARD-002  보완 루프 제어
판단값이 부족하면 추가 질의 요청(ASK_MORE) 결정을 반환해 대화 에이전트(CHAT)가 추가 질의를 생성하게 하고, 도면 객체 선택 또는 보정이 필요한 경우 도면 재확인 요청(REQUEST_OVERLAY_REVIEW) 결정을 반환해 오버레이(OVERLAY) 보정 입력으로 되돌린다.

--- TABLE ---
항목 | 내용
추가 질의 요청 (ASK_MORE) | 판단값 부족 시 CHAT 모듈에 추가 질의 생성을 요청
도면 재확인 요청 (REQUEST_OVERLAY_REVIEW) | 도면 객체 선택·보정이 필요할 때 OVERLAY 보정 입력으로 되돌림
루프 종료 조건 | rule_input_ready=true 도달 또는 사용자가 판단 보류·상담 선택
수용 기준 | 동일 보완 질문이 한 세션 내에서 2회를 초과해 반복되지 않음
--- /TABLE ---

FLOW_GUARD-003  진행·보류·상담 전환 결정
판단값이 충분하면 룰 평가 진행(PROCEED_RULE) 결정을 반환해 법규 룰 엔진(RULE) 평가를 진행하고, 충돌·고위험·불확실성이 큰 경우 보류·상담 전환(HOLD_OR_HANDOFF) 결정을 반환해 판단 보류 또는 상담 전환으로 안내한다. 전체 절차 감시 에이전트(FLOW_GUARD)는 별도 LLM 에이전트로 구성되거나 동일 실행 환경 내 논리 모듈로 구성될 수 있다.

--- TABLE ---
항목 | 내용
룰 평가 진행 (PROCEED_RULE) | 충분성 충족 시 완성된 CommonJudgmentSchema를 RULE 모듈로 전달
보류·상담 전환 (HOLD_OR_HANDOFF) | 결과 충돌·고위험·자동 판단 불가 시 판단 보류 안내 또는 상담 전환
구현 유연성 | 별도 LLM·검증 에이전트로 구성하거나 동일 실행 환경 내 논리 모듈로 구성 가능
수용 기준 | 충돌·고위험으로 분류된 케이스가 100% HOLD 또는 상담 전환 분기로 진입
--- /TABLE ---
