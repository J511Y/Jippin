# TYPOGRAPHY.md — 집핀 타이포그래피·문체 가이드

> **이 문서는 집핀의 폰트 스택·타입스케일·문체 규칙·좋은/나쁜 예의 정본이다.** 진입점은 [`DESIGN.md`](DESIGN.md). 인격·톤 정의는 [`BRAND.md §4 톤앤매너`](BRAND.md). 본 문서는 그 톤을 화면 위에서 어떻게 쓸지 정한다.

## 1. 폰트 스택 (Font stack)

### 1.1 한국어 본문 — Pretendard 우선, 시스템 산세리프 폴백

```css
font-family:
  'Pretendard Variable', 'Pretendard',
  -apple-system, BlinkMacSystemFont, 'Apple SD Gothic Neo',
  'Noto Sans KR', 'Malgun Gothic', system-ui,
  'Segoe UI', Roboto, sans-serif;
```

채택 사유:

- **Pretendard** 는 한국어 본문의 자간·줄높이 균형이 좋고, 영문·숫자 mix 환경(법령 조항·도면 좌표·견적 금액) 에서도 정렬이 안정적이다.
- 미사용 환경(워크스테이션·일부 임베디드 미리보기) 대비 **Apple SD Gothic Neo / Noto Sans KR / Malgun Gothic** 폴백을 준비한다.
- Variable 버전을 우선해 weight 400/500/600/700 를 한 파일로 운반한다.

### 1.2 숫자·코드 — 등폭은 리포트/도면에서만

- 리포트의 견적 금액, 도면 좌표, 코드 블록(예: 법령 조항 번호 정렬) 에 한해 등폭 폴백을 사용한다.

```css
font-family: ui-monospace, SFMono-Regular, 'Cascadia Code',
  'Source Code Pro', Menlo, Consolas, monospace;
```

### 1.3 폰트 로딩 정책

- `font-display: swap` 으로 즉시 폴백 → Pretendard 도착 시 교체.
- 모바일 LTE 환경의 first paint 회귀를 막기 위해 **본문 굵기(400/600) 두 weight 만 우선 로드**, 나머지는 lazy.
- 외부 CDN 의존 대신 self-host 를 권장 (개인정보·자율성). 구체 호스팅 결정은 후속 이슈.

## 2. 타입스케일 (Type scale) — 모바일 우선

본 스케일은 모바일 360-414px 폭을 기준으로 한다. 데스크톱은 동일 단계의 라인 높이를 1.05배 늘리는 것으로 충분하다.

| 토큰 | 용도 | 모바일 size / line-height | weight |
|---|---|---|---|
| `display` | 1순위 결과 헤더 (가능/불가/보류) | 24 / 32 | 600 |
| `h1` | 페이지 헤더 | 22 / 30 | 600 |
| `h2` | 섹션 헤더 (리포트 소제목) | 18 / 26 | 600 |
| `h3` | 카드 헤더, 모달 타이틀 | 16 / 24 | 600 |
| `body` | 본문 기본 | 15 / 24 | 400 |
| `bodyEmph` | 본문 강조 (이름·근거 강조) | 15 / 24 | 600 |
| `caption` | 메타·도움말·캡션 | 13 / 20 | 400 |
| `legal` | 법적 고지 문구 | 12 / 20 | 400 |
| `mono.body` | 견적 금액·도면 좌표·조항 번호 | 14 / 22 | 500 (등폭) |

### 2.1 스케일 사용 규칙

- **히어로 타이포 금지** — 메인 페이지에서도 `display` 보다 큰 단계를 만들지 않는다. 사용자는 마케팅 카피를 읽으러 오지 않는다.
- **결과 화면의 최상단은 `display` 한 줄** — 가능/불가/보류 결론을 한 줄 안에 보여준다. 줄바꿈은 허용하되, 두 줄을 넘기지 않는다.
- **버튼·칩·라벨** 은 한국어 줄바꿈을 고려해 좌우 padding 을 충분히, 폭은 자동에 맡긴다. 1줄 강제 후 ellipsis 금지 (의미 손실 위험).
- **숫자 단독 표기** (금액·면적·조항 번호) 는 `mono.body` 로 정렬한다. 본문 안에서는 일반 본문 weight 를 유지하되 천 단위 구분자(`,`) 를 반드시 표시한다.

## 3. 단락·구조 (Paragraph & structure)

- 한 단락은 한 가지 사실만 담는다. 평균 1~3문장, 모바일에서 4줄 이상 넘기지 않는다.
- 모든 결과 카드는 다음 4구간 구조를 따른다.
  1. **판정(display)** — 가능/불가/보류 + 한 줄 사유.
  2. **근거(body)** — 법령·고시 조항 1~3개, 각 1줄 인용.
  3. **다음 행동(body / button)** — 사용자가 지금 할 수 있는 일.
  4. **법적 고지(legal)** — `notice.legal` 색·`legal` 크기.
- 표는 모바일에서 2열을 초과하지 않는다. 표가 더 필요하면 카드 분할 또는 토글로 접는다.

## 4. 문체 가이드 (Copy guide) — 좋은 예 / 나쁜 예

본 절은 [`BRAND.md §4 톤앤매너`](BRAND.md) 의 인격을 화면 위에서 어떻게 쓸지 보여주는 정본이다. 화면 카피를 쓸 때 반드시 본 절의 좋은 예와 매칭한다.

### 4.1 결과 헤더 (가능)

| 좋은 예 | 나쁜 예 (그리고 이유) |
|---|---|
| 지금 정보 기준으로는 **가능성**이 있습니다. 근거를 확인해 주세요. | "철거 가능합니다 ✅" — 확정 표현·이모지 강조 금지. |
| 제출하신 도면에서 비내력벽 후보가 확인됩니다. | "AI가 분석한 결과 100% 가능합니다" — AI 마케팅·100% 보장 금지. |

### 4.2 결과 헤더 (불가)

| 좋은 예 | 나쁜 예 |
|---|---|
| 제출하신 정보로는 어렵습니다. 사유는 다음과 같습니다. | "불가능합니다." — 사용자 행동 차단형. 사유 없이 단정. |
| 관할 법령(공동주택관리법 시행령 §3 가) 에 따라 제한됩니다. | "행정기관에서 안 해줍니다" — 추측·전언 형식 금지. |

### 4.3 결과 헤더 (보류)

| 좋은 예 | 나쁜 예 |
|---|---|
| 지금 정보만으로는 판단이 어렵습니다. 다음 항목을 추가로 확인해 주세요. | "정보 부족 오류" — 오류로 표현 금지. 사용자의 잘못이 아니다. |
| 도면 일부가 흐려 비내력벽 여부 판단이 어렵습니다. 평면도 한 장을 더 올려 주세요. | "다시 시도하세요" — 무엇을 다시 할지 미지정. |

### 4.4 상담 권장 / CTA

| 좋은 예 | 나쁜 예 |
|---|---|
| 더 자세한 사전검토가 필요하다면 상담을 신청할 수 있습니다. | "지금 신청 안 하시면 손해!" — 공포 마케팅 금지. |
| 인테리어 업체 견적과 비교하실 수 있도록 견적 가이드를 제공합니다. | "최저가 보장!" — 보장 표현 금지. |

### 4.5 법적 고지 (모든 결과 화면 필수)

| 좋은 예 (정본) | 나쁜 예 |
|---|---|
| 본 서비스는 AI 기반 사전 검토 시스템입니다. 최종 행위허가 여부는 관할 행정기관 판단에 따라 달라질 수 있습니다. | "참고용입니다" — 단순화·축약 금지. 정본 문구만 사용. |

> 위 정본 문구는 [`AGENTS.md §4.6`](../../AGENTS.md) 에서 봉인된다. 디자인 단독으로 줄이거나 다듬지 않는다.

### 4.6 시스템 오류 / API 실패

| 좋은 예 | 나쁜 예 |
|---|---|
| 일시적으로 결과를 가져오지 못했습니다. 잠시 후 다시 시도해 주세요. | "오류 발생" — 단독 노출 금지. 다음 행동 미지정. |
| 도면 업로드가 중단됐습니다. 파일을 다시 선택해 주세요. (이용자 잘못이 아닙니다) | "Upload Error 500" — 시스템 코드 노출 금지. |

### 4.7 빈 상태 (Empty state) / 온보딩

| 좋은 예 | 나쁜 예 |
|---|---|
| 주소와 도면을 알려 주세요. 사전검토를 시작하겠습니다. | "환영합니다! 🎉 집핀과 함께 즐거운 인테리어를!" — 톤 위반(이모지·과한 친밀감). |

### 4.8 도면·법령 영역 (전문 보조 축)

| 좋은 예 | 나쁜 예 |
|---|---|
| 평면도에서 비내력벽 후보를 표시했습니다. 색칠된 부분을 눌러 자세히 보세요. | "여기 클릭!" — 위치·맥락 누락. |
| 근거 법령: 공동주택관리법 §35, 시행령 §3 가. | "관련법 참고" — 조항 미지정. |

## 5. 강조 (Emphasis) 규칙

- **굵기(weight)** 와 **색(brand.primary / brand.professional)** 로 강조한다. 이탤릭은 한국어 본문에서 가독성이 떨어지므로 사용하지 않는다.
- 같은 단락 안에서 굵기 + 색 + 밑줄 + 크기를 동시에 쓰지 않는다. 강조는 한 가지만 선택한다.
- 느낌표는 한 화면에 1개 이내. 강조의 도구가 아니다.
- 이모지는 본문에 쓰지 않는다. 단, 시스템 상태 아이콘(앱 자체가 아이콘 라이브러리에서 가져오는 것) 은 예외이며 라벨과 동시에 사용한다 (`COLOR_SYSTEM.md §4.1`).

## 6. 접근성

- 본문은 최소 15px / line-height 1.5 이상.
- 라인 길이는 모바일 35~45자, 데스크톱 60~75자 권장. 그 이상은 줄높이를 더 늘린다.
- 사용자가 시스템에서 글자 크기를 키우면 깨지지 않도록 절대 단위(px) 대신 상대 단위(rem)·또는 Tailwind 의 size 토큰을 사용한다.
- 색 대비는 [`COLOR_SYSTEM.md §6.1 대비`](COLOR_SYSTEM.md) 의 WCAG AA 기준을 따른다.
